# AGENTS

本文件用于说明在本仓库内协作的代码代理、自动化工具或其他贡献者应遵循的项目约束。目标是让文档、实现和后续迭代保持一致。

## 项目定位

- 项目名称：Glosc 2FA
- 项目类型：SwiftUI 应用
- 当前阶段：初始化骨架阶段
- 当前平台：iOS 与 iPadOS 工程已建立
- 目标平台：后续扩展到更完整的 Apple 平台支持

## 当前代码现状

- 应用入口使用 SwiftUI App 生命周期
- 数据层目前仅包含一个示例 SwiftData 模型 Item
- 主界面仍为模板列表页，仅用于验证工程、持久化和基础交互可运行
- 单元测试与 UI 测试仅保留模板代码，尚未覆盖实际业务

任何代理在修改文档或功能时，都必须以这一现状为基准，不得把规划中的能力表述为已经完成。

## 文档约束

- README 负责项目总览、当前状态与路线图
- docs/development.md 负责开发流程、模块拆分建议与测试策略
- 若实现状态发生变化，必须同步更新 README 与开发文档
- 文档描述必须区分“当前实现”“目标能力”“建议方案”

## 开发约束

- 优先实现 OTP 核心链路，不要过早扩展外设或生态功能
- 避免继续沿用模板 Item 作为最终业务模型
- 新增业务逻辑时，应拆分为独立模型、服务或工具类型，不要堆积在 ContentView 中
- 新功能优先补单元测试，再补 UI 层联调
- 不要引入与当前阶段无关的复杂依赖，优先使用 Apple 原生框架

## 推荐架构方向

建议按以下方向演进：

1. Domain
	负责账号实体、OTP 配置、协议枚举、时间步进和验证码生成规则。
2. Data
	负责 SwiftData 持久化模型、仓储封装、导入导出映射。
3. Features
	负责账号列表、账号详情、导入流程、设置页等界面模块。
4. Services
	负责二维码解析、生物识别、剪贴板、同步等系统能力。

在代码量增加前，不要求严格分层目录，但新增代码应朝这个方向靠拢。

## 测试要求

- OTP 生成逻辑必须优先用单元测试覆盖
- URI 解析必须覆盖正常输入、缺失参数、非法算法、非法位数等情况
- UI 测试只覆盖关键主流程，不要用 UI 测试代替业务层测试
- 如果只是文档修改，不需要补测试；如果更改行为，至少说明测试策略

## 平台说明

- 当前 Xcode 工程仍是 iOS 设备家族配置
- README 中提到的 macOS 支持属于产品目标，不应在没有 target 和实现前写成已支持
- 若后续添加 macOS、watchOS 或 Widget Extension，需要同步更新文档与项目结构说明

## 交付标准

- 说明“做了什么”时要可被代码验证
- 说明“计划做什么”时要明确标注为规划或路线图
- 变更尽量小而完整，优先修根因，不做表面占位式改动

## 首选验证方式

- 代码结构变更后，优先检查文档是否仍准确
- 能运行测试时，优先执行对应 target 的测试
- 无法验证的内容，必须在交付说明中明确指出
